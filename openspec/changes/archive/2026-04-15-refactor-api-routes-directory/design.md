## Context

当前 `apps/app/src/api/` 目录采用扁平化结构，所有路由模块文件（10个）直接放在根目录，与 `index.ts`（主入口）和 `middleware/`（中间件）平级。这种结构在路由数量增长时会导致：
- 职责不清晰：无法一眼区分"路由定义"和"其他 API 代码"
- 扩展困难：如果某个路由需要拆分为多个文件（如 handlers、validators），没有自然的子目录

## Goals / Non-Goals

**Goals:**
- 通过创建 `routes/` 子目录，明确区分路由定义和 API 基础设施代码
- 保持所有外部 API 路径不变（`/api/blog`, `/api/contact` 等）
- 保持 TypeScript 编译通过，无类型错误

**Non-Goals:**
- 不修改任何路由逻辑或业务代码
- 不改变中间件的位置或结构
- 不修改 API 响应格式或错误处理
- 不涉及前端调用代码的修改（前端使用相对路径 `/api/*`，不受影响）

## Decisions

### 1. 目录结构选择

**决策**: 采用 `api/routes/` 平级结构，而非按功能域分组

```
api/
├── index.ts           # 主入口 + 全局配置
├── middleware/        # 横切关注点
│   └── cache.ts
└── routes/            # 所有路由定义
    ├── blog.ts
    ├── comments.ts
    └── ...
```

**理由**:
- 简单直接，改动最小
- 10 个路由文件数量适中，无需进一步分组
- 为未来扩展预留空间（可在 routes/ 下创建子目录）
- 符合单一职责原则：routes/ 专注路由定义

**备选方案**:
- 方案 B（按功能域分组）：过早优化，当前每个路由只有一个文件
- 方案 C（保持扁平 + 命名约定）：目录仍然混合，不够清晰

### 2. 中间件位置

**决策**: `middleware/` 保持与 `routes/` 平级

**理由**:
- 中间件是横切关注点，可能被多个路由共享
- 语义上属于 API 基础设施，而非某个具体路由
- 相对路径清晰：`../middleware/cache`

### 3. index.ts 职责

**决策**: 保持 `index.ts` 在 `api/` 根目录，职责不变

**理由**:
- 作为 API 入口点，位置符合直觉
- 包含全局错误处理和健康检查，属于应用级配置
- 外部调用点（`app/api/[[...route]]/route.ts`）无需修改

## Risks / Trade-offs

### Risk 1: 相对路径变长
- **影响**: 路由文件中引用 middleware 的路径从 `./middleware/cache` 变为 `../middleware/cache`
- **缓解**: IDE 自动补全可避免路径错误，TypeScript 编译器会验证

### Risk 2: Git 历史记录
- **影响**: 文件移动会导致 `git blame` 历史中断
- **缓解**: 使用 `git mv` 或 `git log --follow` 可追踪历史

### Trade-off: 目录深度增加
- **缺点**: 访问路由文件需要多一层目录
- **优点**: 职责清晰，IDE 文件树更易导航
- **结论**: 对于 10+ 个路由文件，清晰的目录结构优于扁平化
